
#### Verification Flows for *Amadeus Board the Flight*

##### Overview and Episode Scope

The *Board the Flight* use case is structured around **two distinct verification episodes**, each aligned with its operational context and communication constraints:

- **Episode 1 – Online verification at Self‑Service Online Check‑In (SSCI)**  
  This is a **remote, online flow**, based on an **OpenID4VP‑style interaction**. It is designed to be compatible with the **DC API** where the EUDI Wallet implementation supports it.

- **Episode 2 – Proximity verification at the airport**  
  This is a **proximity‑based flow**, primarily aligned with **ISO/IEC 18013‑5**, for short‑range, in‑person verification at boarding.

Across both episodes, the **EUDI Wallet** is the common presentation channel, while Amadeus and airline systems apply domain‑specific verification and eligibility checks.

---

**Episode 1 – Collect and Verify API at Self‑Service Online Check‑In (SSCI)**

Episode 1 covers **Advance Passenger Information (API)** collection and verification in an **online context**, before the traveler arrives at the airport.

**Verification Pattern**

The flow follows an **OpenID4VP‑style pattern**:

1. The airline or an Amadeus backend system acting as **verifier** initiates a request for API‑relevant attributes.
2. The request is delivered to the traveler’s **EUDI Wallet** via a supported online engagement method (redirect, deep link, or QR).
3. The wallet displays:
   - The requested *digital credential* (e.g. DTC DG1 or PID attributes)
   - The purpose of the request (API collection)
   - The minimum data to be disclosed
4. The traveler authenticates locally to the wallet and **consents** to disclosure.
5. The wallet constructs a **verifiable presentation** and returns it to the verifier (directly or via an intermediary service).
6. The verifier performs:
   - Issuer trust and signature validation
   - Validity and status checks
   - Policy checks specific to API requirements
7. The verified API data is forwarded to airline and border authority systems, and the check‑in process continues or fails accordingly.

##### Key Characteristics

- Fully **remote and online**
- **No biometric capture** required
- Credential usage is limited to **biographic and document data** (e.g. DG1 or PID)
- Designed to reduce manual data entry and improve API data quality

---

**Episode 2 – Board the Flight at the Airport**

Episode 2 covers **in‑person identity and entitlement verification** at the airport, most notably at the **boarding gate**.

**Verification Pattern**

This episode primarily uses a **proximity‑based verification flow**, aligned with **ISO/IEC 18013‑5**:

1. The boarding system prompts the traveler to present their **EUDI Wallet**.
2. A **proximity engagement** is established (typically via QR code or NFC).
3. The verifier requests a *digital credential* (e.g. DTC DG1, and DG2 if biometric mode is enabled).
4. The wallet displays the request and purpose (boarding verification).
5. The traveler authenticates locally to the wallet and approves disclosure.
6. The wallet transmits the **verifiable presentation** over the proximity channel **using bluehtooth** instead of internet in this **ISO/IEC 18013‑5** protocol .
7. The verifier performs layered checks:
   - Cryptographic integrity and issuer trust
   - Credential validity and status
   - Subject binding between identity and boarding entitlement e.g verify boarding pass validity in backend system
   - Operational checks against airline systems (flight, boarding status)

##### Optional Biometric‑Enhanced Mode

- If biometric is enabled in digital credential, a **one‑off live facial capture** may be triggered.
- The check remains **1:1 verification**, not identification.
- The live capture is compared against:
  - A protected biometric reference derived from the credential ( PID or DTC)
- No biometric data is stored after the transaction.

##### Key Characteristics

- **Proximity‑based** interaction
- Optimized for **traveler user experience** e.g no internet required, no additoinal applicaiton to be downloaded, no pre-registration step, **"your EUDIW is our pass"**

---

##### Summary

- **Episode 1 (SSCI)** focuses on **online OpenID4VP‑style verification** for API collection, with optional DC API compatibility.
- **Episode 2 (Airport boarding)** focuses on **proximity‑based ISO/IEC 18013‑5 verification**, typically using Bluetooth.
- Both episodes reuse the same **EUDI Wallet**, while applying different verification logic and policies suited to their operational context.
