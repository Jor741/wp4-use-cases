## Relation to Other Work Packages and Deliverables

WP4 is one of 8 Work Packages in APTITUDE, and it cannot be understood, or delivered, in isolation. The European Digital Identity Wallet ecosystem, by design, is cross-sectoral: a single wallet holds credentials issued by many different authorities and is presented to many different relying parties. A traveller who uses their wallet to board a flight may, within the same trip, present Digital Travel Credentials at a border, pay for a taxi, check into a hotel, and access their university campus. Each of these interactions may touch one or more different APTITUDE Work Packages. For WP4 to deliver its end-to-end user journeys, it must align with the other Work Packages that contribute the underlying capabilities.

Three forms of coordination drive the relationships documented in this section:

* **Shared technical standards.** All APTITUDE use cases must comply with the same Architectural Reference Framework (ARF) profiles, the same cryptographic standards, and the same trust infrastructure. This alignment is mainly the responsibility of WP2, which acts as the technical backbone for the whole project.
* **Credential overlap.** Several WP4 use cases rely on credentials that are issued or governed by other Work Packages. Passport-derived credentials come from WP3 (DTC). Payment credentials come from WP6 (PBEE Framework). Where such overlap exists, defining interface between Work Packages will help to consistently handle the credential lifecycle (issuance, presentation, revocation).
* **Combined user journeys.** Many of the scenarios piloted in WP4 extend across multiple domains. A cross-border train journey followed by a hotel stay, for example, may involve a student-fare ticket, a payment authentication, and a hotel booking confirmation, within the same wallet. These combined journeys require the correct interfacing between the Work Packages that own each touchpoint, so that the user experience is seamless rather than fragmented.

### The Work Package landscape of APTITUDE

Before focusing on the three Work Packages with which WP4 has to collaborate more directly, it is useful to place WP4 within the full set of APTITUDE Work Packages:

* **WP1 (Project Management and Coordination)** provides the overall governance structure, risk management, and reporting mechanisms to which WP4 contributes. D1.1 (Roadmap for the implementation of the ecosystem) and D1.2 (Sustainability Business Strategy) provide the strategic framing within which WP4 operates.
* **WP2 (Technical Standards and Specifications)** defines the common technical profiles, implements and runs the interoperability testbed, and provides the trust infrastructure. WP4 implementations must comply with the outputs of WP2.
* **WP3 (Wallet for Travel, DTC)** develops the Digital Travel Credentials that several WP4 travel use cases rely on.
* **WP5 (mobile Vehicle Registration Certificate)** pilots an attestation that is adjacent to WP4, without directly interacting with it. The two Work Packages share the common technical foundation provided by WP2 and may exchange lessons learned.
* **WP6 (Wallet for Payment and Banking)** develops the Payment and Banking EUDIW Extension (PBEE) Framework, on which WP4 use cases involving payment depend.
* **WP7 (Compliance, European Values and Civil Society)** ensures that the project as a whole respects ethics, compliance, and European values. WP4 use cases that collect or process real personal data feed into the compliance assessments produced by WP7.
* **WP8 (Marketing, Communication and Dissemination)** disseminates project results, including WP4 important findings and milestones, to external stakeholders.

As a conlusion, the Work Packages with direct operational dependencies on WP4, and for which interface definitions are documented in this deliverable, are WP2, WP3, and WP6. Each is treated in following dedicated subsections.

### Alignment with the stock-taking phase across Work Packages

It is worth noting that D4.1 does not exist in isolation within the current project phase. The stock-taking and specification activity that produced D4.1 has parallel "twins" in the other use case-based Work Packages: WP3 is producing D3.1 (Stock-taking, Analysis and Specifications) through Task T3.2; WP5 is producing D5.1 (UC Specification and Analysis Results) through Task T5.1; and WP6 is producing D6.1 (Sprint/PI Plan and filled-in Backlog) through its inception phase (T6.1) following an agile approach. Although each Work Package addresses a separate domain, the stock-taking activities are deliberately aligned: they all examine existing processes, assess legal constraints, and define functional specifications that must ultimately comply with the common technical standards and interoperability profiles established by WP2. Where WP4 use cases are within Work Package boundaries, as is the case for travel scenarios that depend on WP3 Digital Travel Credentials or WP6 payment capabilities, the respective stock-taking phases have been coordinated so that interface assumptions and credential requirements remain consistent across the deliverables that emerge from this phase.
