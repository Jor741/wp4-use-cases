## 3.4 Use Case Specification Template

This section presents the standardised template that was used by all 11 UC leads to document their use cases. The purpose of the template is to ensure that each use case is described at a comparable level of detail and along the same structural lines, so that readers can move from one UC section to another (or from one UC specification sheet in Annex A to another) without having to re-orient themselves to a different layout. The template was designed by the WP4 coordinator (GRNET) during the stock-taking phase, in close dialogue with a representative subset of UC leads, and was iteratively refined as the first completed UC sheets surfaced fields that needed clarification.

### Why a standardised template

Before the template was introduced, the 11 use cases were captured only in the lightweight spreadsheet described in Section 3.1. The spreadsheet was sufficient to give the consortium a shared view of the UC portfolio and to surface dependencies between use cases, but it was not sufficient to support the technical and legal analyses that D4.1 must deliver. A more structured specification tool was therefore needed.

A standardised template offered several advantages over the alternative of letting each UC lead invent its own documentation style:

* **Comparability.** A reader of D4.1 can compare UC 1 (air travel) and UC 10 (hotel booking) along exactly the same dimensions, even though the two use cases operate in different sectors and involve very different partners.
* **Completeness.** The template defines a minimum set of fields that every UC specification must address, so that no UC sheet is missing essential information (such as the unhappy paths or the list of applicable regulations).
* **Traceability.** Because every UC is described in the same fields, the content of D4.1 can be traced forward into D4.2 (implementation manual) and D4.3 (validation report), which will address the same UCs along the same dimensions.
* **Reuse across work packages.** The same template structure is aligned, as far as practicable, with the UC documentation approaches used by WP3 and WP5 in their own stock-taking deliverables, so that a reader moving between D3.1, D4.1, and D5.1 encounters a familiar vocabulary.

### Goals of the template

The template was designed with three specific goals in mind:

1. To **enforce consistency** of structure across the 11 use cases, while guiding the content to avoid a rigid shape that wouldn't
accommodate the diversity of sectors covered by WP4.
2. To **guide UC leads** through the information that D4.1 requires, by providing, for each field, a short description of its purpose together with an exemplary value drawn from a representative UC. In this way, a UC lead can fill the template with reasonable confidence in what is expected.
3. To **produce a specification that is actionable in D4.2**, in the sense that every field that an implementation team will need to refer to when integrating a UC is captured, so that the implementation manual can build on the template rather than re-gathering information.

### Structure of the template

The template is organised into four groups of fields: metadata, descriptive fields, credential and flow fields, and completeness fields. Each field is introduced below with a short description of its purpose and, where helpful, an exemplary value.

#### Metadata fields

These fields identify the use case and position it within the WP4 portfolio.

* **UC ID.** The numeric identifier of the use case, in the range 1 to 11. Example: `UC 2`.
* **UC Title.** The human-readable title of the use case, as agreed within the WP4. 
* **UC Lead.** The partner organisation(s) responsible for the use case. Where a use case has more than one lead, all co-leads are listed.
* **Supporting Partners.** The full list of APTITUDE partners that contribute to the use case in any role (credential issuer, relying party, wallet provider, technical integrator, legal advisor). 
* **Scenarios Covered.** The WP4 scenarios to which the use case contributes (Streamlined Travel, Hotel Check-In, Student Access). A use case may contribute to more than one.
* **Member States Involved.** The EU Member States (and, where relevant, non-EU countries) in which the use case is piloted. 

#### Descriptive fields

These fields narrate the use case in business-level terms, accessible to non-technical readers.

* **UC Summary.** A concise paragraph that captures what the use case does, who it is for, and why it matters.
* **User Story.** One or more short user stories in the canonical form *"As a [role], I want to [action] so that [benefit]"*. The user story makes explicit the value delivered to the end user.
* **Actors.** The list of parties that take part in the use case, with their role in EUDIW terms. Typical actors include the **wallet holder** (end user), the **wallet provider** (Member State issuer), the **PID provider**, the **credential issuer** (for sector-specific EAAs), the **relying party** (verifier), and any **trust service provider** involved in the trust chain. Each actor is described in one sentence that states what they contribute to the flow.
* **Context and Preconditions.** The situational setup that must hold for the use case to apply, and the preconditions that must be satisfied before the flow can start. Example preconditions: *"The user has an activated EUDIW with a valid PID"*; *"The user has a booking reference issued by the airline"*.

#### Credential and flow fields

These fields describe the technical substance of the use case: the credentials exchanged and the steps of the flow.

* **Credentials Involved.** The full list of credentials that appear in the use case, each described by its type (PID, EAA, QEAA, DTC), its issuer, its verifier, and its data model (mdoc, SD-JWT). The credentials referenced here align with the common technical foundations described in Chapter 7 and avoid duplicating content already covered there.
* **User Journey.** A step-by-step narrative of what the end user does and sees, from the start of the scenario to its conclusion. The journey is written in plain language (no protocol names, no data structure references) so that it can be read by non-technical stakeholders. It is typically illustrated with a flow diagram authored in Mermaid, following the diagrammatic standards of Section 3.4.1.
* **Technical Flow.** A sequence-level description of the message exchanges between the wallet, the credential issuers, and the relying parties. This is the technical counterpart of the user journey and uses the protocol names (OpenID4VCI, OpenID4VP) and credential formats defined in Chapter 7. It is typically illustrated with a Mermaid sequence diagram.

#### Completeness fields

These fields ensure that each UC specification contains the information that D4.2 and D4.3 will need.

* **Unhappy Paths.** The situations in which the happy-path flow does not apply and what the system is expected to do in those cases. Examples include an expired credential, a lost or replaced wallet, a cancelled booking, an offline verification point, and a mismatch between the name on the ticket and the name on the PID. Each unhappy path is described together with the expected behaviour and, where relevant, the recovery path.
* **Functional Requirements.** The list of things that the implementation must be able to do, expressed as short imperative statements. Example: *"The wallet shall support the OpenID4VP proximity profile"*.
* **Non-functional Requirements.** The quality attributes that the implementation must exhibit (performance, offline capability, accessibility, security). 
* **Applicable Regulations.** The list of legal and regulatory references that constrain the use case, in addition to the baseline framework established by eIDAS 2.0. This field feeds into the analysis of Chapter 5.
* **Dependencies.** The dependencies of the use case on other use cases within WP4 (for example, shared credentials) and on other Work Packages (for example, WP3 for DTC issuance or WP6 for payment steps). This field feeds into the cross-WP coordination work documented in Section 2.3.
* **KPIs / Success Criteria.** The indicators that will be used in D4.3 to evaluate whether the use case met its objectives during piloting. Example: *"At least 80% of test users were able to complete the ETA application entirely within the wallet, without falling back to the legacy web form"*.
* **Open Issues.** The items that remain unresolved at the time of D4.1 and that will require attention before or during the implementation phase. This field feeds into Section 10.2 (Open Issues and Outstanding Decisions).

Chapter 4 of this deliverable presents, for each of the 11 use cases, the content of the template in narrative form. 

Several fields of the template call for diagrams, notably the user journey and the technical flow. The diagrammatic standards that apply to these diagrams (choice of diagram type, colour coding, orientation, labelling conventions, and the choice of Mermaid as the single diagramming language) are documented in mermaid.
