#### User Consent
**User consent** is the legal authorisation that allows a relying party to process personal data disclosed from the EUDI Wallet.

In the *Amadeus Board the Flight* use case, user consent ensures that:
- Data sharing is **explicitly approved by the traveler**
- Processing is **purpose‑specific and transparent**
- Legal obligations under **GDPR and eIDAS 2.0** are met

User consent answers the question:

> *Why is this data allowed to be shared and used?*

## Legal Perspective

From a legal standpoint, user consent:

- Is a **lawful basis for processing** when required
- Must be:
  - Freely given
  - Specific
  - Informed
  - Unambiguous
- Must be **explicit** when biometric data is involved (e.g. DG2)

Critically:
- Consent is **transaction‑specific**
- Consent is **not transferable** across episodes
- Consent must be **demonstrable and auditable**

Wallet installation, possession, or authentication **does not constitute consent**.

## Consent vs Wallet Authentication

User consent must be clearly distinguished from wallet authentication.

### Wallet Authentication
- Performed using PIN or device biometrics
- Purpose: security and access protection
- Legal meaning: **none for data sharing**

### User Consent
- Performed via an explicit approval action
- Purpose: authorise disclosure and processing
- Legal meaning: **lawful authorisation**

> Unlocking the wallet ≠ consenting to data disclosure.

## Implementation Perspective

### Explicit Consent Action

User consent is implemented as:
- A deliberate **approve / confirm action**
- Performed after transparency is shown
- Binding:
  - Relying party identity
  - Declared purpose
  - Exact disclosed data set

This action generates a **verifiable consent event**.

## Single Screen Principle

In the EUDI Wallet:
- **Selective disclosure and consent are presented on the same screen**
- But they remain **conceptually and legally distinct**

On that screen, the traveler can see:
- What data will be shared (selective disclosure)
- Who requests it and why (consent context)

A single tap confirms both **data scope** and **legal authorisation**.

## Application to *Amadeus Board the Flight*

### Episode 1 — Self‑Service Online Check‑In (SSCI)

- **Consent scope:** API collection
- **Disclosed data:** DG1
- **Biometric data:** none
- **Consent type:** explicit approval

### Episode 2 — Airport Boarding

- **Consent scope:** boarding verification
- **Disclosed data:** DG1 + DG2
- **Biometric data:** DG2 (facial image)
- **Consent requirement:** **explicit consent mandatory**
- **Processing:** one‑off 1:1 biometric verification

Consent given in Episode 1 does **not** apply to Episode 2.

## Key Characteristics of User Consent

- Legal authorisation, not a technical filter
- Purpose‑bound and time‑bound
- Explicit affirmative action required
- Auditable and revocable

> User consent authorises **why and under which conditions** personal data may be processed, independently from which data is technically disclosed.
